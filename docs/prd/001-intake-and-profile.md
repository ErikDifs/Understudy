# PRD 001 — Intake Chat and Profile Synthesis

## Problem Statement

The Understudy currently attends meetings and produces generic summaries. It knows nothing persistent about the user beyond the free-text instructions entered at the time of meeting delegation. This means:

- Summaries are not personalized to what the user actually cares about
- There is no basis for escalation detection — the bot cannot distinguish a routine mention of budget from a real decision that needs the user
- Every meeting delegation starts from zero; the Understudy never gets smarter about who it's representing

The user needs their Understudy to actually know them — their work, their constraints, what topics matter, what has already been settled — so it can represent them faithfully and only interrupt when something genuinely new comes up.

---

## Solution

An in-app Intake chat where the user converses with their Understudy to build a persistent Profile. The Understudy asks probing questions across multiple sessions — what the user is working on, what decisions are already settled, what triggers them to care — and after each session synthesizes a dense Profile document that gets carried into all future meeting contexts. Meeting summaries immediately become more personalized as a result.

---

## User Stories

1. As a user, I want a dedicated page where I can talk to my Understudy, so that I can teach it who I am without having to fill out a form.
2. As a user, I want my Understudy to ask me probing questions rather than wait for me to know what to share, so that the Profile gets built even when I don't know where to start.
3. As a user, I want the Understudy to follow up on what I say — dig deeper, clarify, challenge — so that the Profile captures nuance, not just surface-level facts.
4. As a user, I want the conversation to feel natural and back-and-forth, so that Intake doesn't feel like a chore.
5. As a user, I want the Understudy to have a consistent persona and voice in Intake, so that it feels like talking to a specific agent that represents me, not a generic chatbot.
6. As a user, I want my Understudy to remember what I told it in previous Intake sessions, so that each new session builds on the last rather than starting from scratch.
7. As a user, I want to be able to return to the Intake chat at any time to update or deepen my Profile, so that the Understudy gets better as my work evolves.
8. As a user, I want to see what my current Profile says, so that I know what my Understudy knows about me and can spot gaps.
9. As a user, I want the Profile to update after each Intake session, so that every conversation I have immediately makes the Understudy more accurate.
10. As a user, I want Intake responses to stream in real time, so that the chat feels responsive and alive rather than making me wait for a full reply.
11. As a user, I want the Intake chat to work on my phone, so that I can do a session while I'm away from my desk.
12. As a user, I want meeting summaries to reference my specific concerns and context, so that they're actually useful and not generic recaps.
13. As a user, I want the Understudy to use what I told it in Intake when summarizing a meeting — even if I gave no per-meeting instructions — so that the default output is already personalized.
14. As a user, I want the Understudy to flag things in a summary that directly relate to my stated priorities, so that I can scan the summary and immediately see what matters to me.
15. As a user, I want a meeting summary to omit or de-emphasize topics I've already told the Understudy are settled, so that noise is filtered out.
16. As a user, I want the Intake chat to be keyboard accessible, so that I can navigate and submit without a mouse.
17. As a user, I want the Intake chat to use clear, readable contrast and legible text, so that it's accessible regardless of my vision.

## Implementation Decisions

### Schema

Two new tables:

**`intake_messages`** — the raw conversation history between the user and their Understudy. Each row has a role (`user` or `assistant`), the message content, and a timestamp. This is the source of truth for the Profile; it is never deleted.

**`profiles`** — a single row (single-user for now) containing the `synthesized_document`: a dense natural language summary generated by Claude from the full Intake history. Also stores `updated_at` so the user can see when their Profile was last refreshed.

No `user_id` foreign keys yet — single-user schema by design (ADR-005). The schema is structured to receive them later without breaking changes.

### Chat API

A new `POST /api/intake` endpoint:

- Accepts the user's latest message as the request body
- Persists it as a `user` role row in `intake_messages`
- Fetches the full conversation history from the DB
- Calls Claude with the full history as the `messages` array (stateless on the server; the DB is the state)
- Streams the response back to the client
- Once streaming is complete, persists the assistant's response as an `assistant` role row in `intake_messages`
- Triggers Profile synthesis as a fire-and-update (does not block the response)

### Understudy system prompt

The Intake system prompt establishes the Understudy persona: it knows it is a specific person's AI representative, its job in this conversation is to learn enough to represent that person faithfully in meetings, and it should be a curious, probing interviewer rather than a passive recipient. It asks follow-up questions, challenges vague answers, and pushes for specifics: not "what do you work on" but "what decision would derail you if it went the wrong way this month."

### Profile synthesis

Triggered server-side after each assistant turn (fire-and-update, non-blocking). The synthesis prompt sends the full `intake_messages` history to Claude and asks it to produce a dense natural language document that captures: what the user is building, what they care about in meetings, what is already settled and need not be escalated, and what would cause them to want to be pulled in. The result overwrites `profiles.synthesized_document` and updates `profiles.updated_at`.

### Profile injection into meeting summarization

The existing `summarizeMeeting` function is updated to fetch the current `synthesized_document` before calling Claude. It is prepended to the system prompt as "What you know about the user:" so the model has full context when writing the summary. If no Profile exists yet, the function falls back to the current behavior.

### UI

A new `/profile` route with two sections:
- A read-only display of the current `synthesized_document` (or a prompt to start Intake if none exists yet)
- The Intake chat interface: a scrollable message history and a text input that submits via the `POST /api/intake` endpoint, streaming the response into the UI

The chat input supports submit-on-Enter (with Shift+Enter for newline). The message list auto-scrolls to the latest message. The send button is disabled while a response is streaming. The layout is mobile-first: single column, full-width input, comfortable tap targets.

## Testing Decisions

A good test for this feature tests observable external behavior, not internal implementation. It does not mock Claude — the quality of the Understudy's responses and the accuracy of Profile synthesis are the things worth verifying, and those require a real model call.

**The testing seam is `POST /api/intake`.** It is the single entry point for the entire feature — chat, synthesis, and eventual Profile injection all flow from it.

**What to test:**

- Send a sequence of Intake messages covering a user's specific problem, their constraints, and a settled decision (e.g. "we agreed the budget is fixed at €50k"). After the sequence, read the `profiles` table and assert the `synthesized_document` mentions the settled budget. This confirms the synthesis pipeline is working end-to-end.
- After the Profile is set, trigger a meeting summarization against a transcript that mentions budget. Assert the summary de-emphasizes or notes the budget as already known. This confirms Profile injection into summarization.
- Send a single opening message and assert the assistant response contains a follow-up question (not a statement). This confirms the Understudy is behaving as a probing interviewer, not a passive responder.

**Prior art:** the existing `summarizeMeeting` function is the closest precedent — it calls Claude with a full prompt and asserts on the returned content. The same pattern applies here, extended to a multi-turn conversation.

No unit tests on internal helper functions (prompt construction, synthesis trigger) — these are implementation details. All assertions are on DB state or API response content.

## Out of Scope

- **Escalation detection** — a separate feature; depends on this Profile foundation but is not part of it
- **Meeting-based Intake** (bot joins a call to conduct the interview) — deferred (ADR-001)
- **Auth and multi-user** — deferred (ADR-005); no `user_id` FKs in this work
- **Realm and networking** — later phase; depends on multi-user
- **Profile versioning or history** — the synthesized document is overwritten on each synthesis; no history of previous Profile versions
- **Manual Profile editing** — the Profile is derived from Intake, not hand-authored
- **Deleting or resetting Intake history** — not addressed in this iteration

---

## Further Notes

The Profile synthesis is the most fragile piece — it depends on the quality of the Intake conversation and the prompt. The right approach is to iterate on the synthesis prompt against real Intake transcripts before treating it as stable. Keep the synthesis prompt separately maintained and easy to tune without touching application logic.

The Understudy's persona in Intake sets the tone for everything that follows — the meeting bot, the networking agent, all future contexts share this voice. Getting the Intake system prompt right is more important than it looks.
